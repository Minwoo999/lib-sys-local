# 📕 도서 관리 시스템 상세 설계 

---

## 1. 도메인 규칙
> 이용 제약 사항

- **한도:** 1인당 최대 `5권`
- **기간:** 대출일 포함 `7 + 7일`. (반납일이 휴일일 경우 다음 영업일로 연장, 연장 신청시 7일 추가)
- **제한:** `연체 중`이거나 `계정 정지` 상태인 회원은 트랜잭션 처리.



## 2. 핵심 로직 플로우 

### 2-1. 대출
### 2.1 성공 시나리오 
1. 사용자가 `대출 버튼` 입력
2. `로그인 여부 확인`
3. **비관적 락(Pessimistic Lock)**
3. 회원 대출 가능 여부 검증 (권수 체크, 상태 체크).
4. 도서 잔여 권수(`current_cnt`)가 1 이상인지 확인.
5. `Transaction` 시작:
    - `LOAN` 테이블 레코드 생성.
    - `BOOK` 테이블 `current_cnt` -1.
    - `MEMBER` 테이블 `current_loan_cnt` +1.
6. 완료 메시지 반환.

### 2.2 예외 처리 (Exception Handling)
| 에러 코드 | 상황 | 응답 메시지 |
| :--- | :--- | :--- |
| `ERR_LIMIT` | 5권 초과 대출 시도 | "최대 대출 권수를 초과했습니다." |
| `ERR_OVERDUE` | 미반납 연체 도서 존재 | "연체 도서를 먼저 반납해 주세요." |
| `ERR_EMPTY` | 책 재고가 없을 경우 | "대출 가능한 재고가 없습니다." |


### 2-2. 반납

### 2.1 성공 시나리오
1. 사용자가 `반납 버튼` 입력
2. `사용자 정보 확인`
3. `반납일`이 지났으면 지난 날짜만큼 대출불가
4. 도서 잔여 권수(`current_cnt`)가 1 이상인지 확인.
5. `Transaction` 시작:
    - `LOAN` 테이블 레코드 수정.
    - `BOOK` 테이블 `current_cnt` +1.
    - `MEMBER` 테이블 `current_loan_cnt` -1.
6. 완료 메시지 반환.

## 3. 기술적 상세 (Technical Details)
###  동시성 제어 전략
- **방식:** 비관적 락 (Database Level Lock)
- **이유:** 학습용 프로젝트이나, 한정된 수량의 도서에 대해 데이터 무결성이 최우선이므로 `FOR UPDATE`를 사용하여 충돌 방지.
- **타임아웃:** DB 커넥션 점유 방지를 위해 최대 5초 대기 후 실패 처리.




## 4. 조회 및 페이징 (Search & Paging)

---

### 4-1. 페이징 규칙
- **방식:** 오프셋 기반 페이징 (Offset-based Pagination)
- **사이즈:** 페이지당 기본 `10권`
- **정렬:** 최신 등록순 (`created_at DESC`) 및 도서명 가나다순 (`title ASC`)

### 4-2. 로직 플로우
1. 클라이언트로부터 `page`(페이지 번호)와 `size`(페이지당 개수) 수신.
2. **Total Count 조회:** 검색 조건에 부합하는 전체 레코드 수 계산.
3. **Offset 계산:** `(page - 1) * size` 공식 적용.
4. **Data 조회:** `LIMIT`과 `OFFSET`을 사용하여 해당 구간의 도서 목록 추출.
5. 데이터와 메타데이터(전체 페이지 수 등)를 포함하여 응답.

### 4-3. 기술적 상세

**API 응답 구조 (Response DTO)**
```json
{
  "items": [],           // 도서 목록 배열
  "totalElements": 125,  // 전체 아이템 개수
  "totalPages": 13,      // 전체 페이지 수
  "currentPage": 1,      // 현재 페이지
  "size": 10             // 페이지당 개수
